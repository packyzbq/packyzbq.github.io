CNCF 生态周报 2020-01-21

Posted on January 21, 2020

这个系列是我一直想做的系列，由于现在的工作需要关注CNCF社区，修改K8s源码，因此需要持续关注社区。本篇主要是基于 https://kubeweekly.io/2020/01/17/kubeweekly-199/ 这篇文章的总结和扩展。

要闻

Falco 进入 CNCF Incubator 项目

Falco是开源云原生运行时安全项目，它是事实上的Kubernetes威胁检测引擎。Falco检测到意外的应用程序行为，并在运行时发出威胁警报。

这个项目在当前关注的容器安全问题填补相关空缺。

Prometheus 2.15.2 发布

本周 Prometheus 发布了 v2.15.2 版本，其修复了两个 TSDB 相关的问题。

修复对 2.1.0 之前版本构建的 TSDB 块的支持，这个问题直接影响的是那些直接从 2.1.0 版本之前直接升级到 2.15 的用户，根本原因是在 2.1.0 版本加入的一个对 key 排序的特性；
修复了 TSDB 在 Windows 下的压缩问题；

Release Note

istio 1.4.3 发布

这个版本修复了诸多bug，并添加了部分提升：

  • Fix

    • 修复了注入相关的模板。当 POD 有多个 container 但 container 未暴露端口时，istio-proxy 无法启动的问题。#18594
    • 修复了 Mixer 为 secret 创建大量 watcher 可能导致 Kubernetes api-server OOM 的问题 #19481

  • Imporve:

    • 注入模板目前支持 PSP；#17318

其他信息见 Release Note

上游进展

LoadBalancer Service 支持多种不同 protocol 类型的 port

目前一个 LoadBalancer Service 可以写多个 port，但是这些 port 的类型必须相同，比如都是 tcp 或 udp。这个 PR 允许在一个 LoadBalancer Service 中定义多种不同类型的 port，以支持不同云厂商提供的 service 服务 #75831

API Server 现在需要 client 声明其协议版本

#86646

项目推荐

Grafana lab 开启新工程 – Tanka

Tanka是Kubernetes的可组合配置实用程序。它利用Jsonnet语言来实现灵活，可重用和简洁的配置。

这个项目是可以替代 helm 实现应用部署的，主要解决 helm 使用 YAML 模板部署时的诸多问题，

  • YAML 模板较为脆弱：依赖缩进，模板引擎不识别 YAML 语法，只替换对应的参数（只替换 .Values.xxx）
  • 不易扩展：chart 模板开发者没有暴露某些参数，其他人则需要 fork 整个chart模板进行修改
  • 抽象性差：Helm不允许构建比替换字符串更高级别的抽象。
  • YAML 较为复杂

Chaos Mesh

在分布式计算的世界中，群集随时随地都可能发生故障。传统上，我们使用单元测试和集成测试来确保系统已准备就绪，但是随着集群规模，复杂性数量和数据量（基于PB级别）的增加，这些测试仅涵盖冰山一角。为了更好地识别系统漏洞并提高弹性，Netflix发明了Chaos Monkey，并将各种类型的故障注入基础架构和业务系统。这就是Chaos Engineering的起源。

Chaos Mesh是一个通用的Chaos Engineering（混沌工程）平台，具有针对Kubernetes上复杂系统的全方位故障注入方法，涵盖Pod，网络，文件系统甚至内核中的故障。由 TiDB 团队开发。

github: https://github.com/pingcap/chaos-mesh
文章链接: https://pingcap.com/blog/chaos-mesh-your-chaos-engineering-solution-for-system-resiliency-on-kubernetes/

Vitness

Vitess是一个用于部署，扩展和管理大型开源数据库实例集群的数据库解决方案。它目前支持MySQL和MariaDB。它在架构上可以像在专用硬件上一样有效地在公共或私有云架构中运行。它结合并扩展了许多重要的SQL功能以及NoSQL数据库的可伸缩性。

github: https://github.com/vitessio/vitess

kubernetes message queue

适用于Docker和Kubernetes的企业级消息代理。在可用于生产的集群中交付，并针对任何类型的工作负载而设计。KubeMQ作为小型轻量级Docker容器提供，专为Kubernetes或支持Docker的任何其他容器编排系统中运行的任何工作负载和架构而设计。

这个是基于 K8s 原生开发，未来可能会集成到 K8s 中，按照这个趋势，未来可能会有更多的中间件汇集成到 K8s中。

github: https://github.com/kubemq-io
