K8s的主要架构组件分为两个部分：

  • master主节点
  • node 工作节点

其中在 master 主节点中，包括：

  • etcd（用于存储 K8s 集群的信息，包括 pod deployment 等）
  • API 服务器
  • 调度器
  • node controller

node 节点包括：

  • kubelet
  • kube proxy
  • 容器运行时

节点是承载 K8s 的资源，因此我们先从 K8s 对节点的管理说起

Node

node 状态包含一下信息：

  • 地址： 包括域名（kubelet –hostname-override 修改），外部Ip，内部ip
  • 状态： Ready OutofDisk等
  • 容量：
  • 信息

node 状态

当一个节点处于 非 Ready 状态超过 pod-eviction-timeout，节点控制器会向kube-controller-manager传递参数，来驱逐已经调度到当前节点的 pod。（默认时间是 5 min）

某些情况下，节点无法通信（api server 与节点的 kubelet 失联），pod 无法立即删除直到重新连接，这段时间里，pod 仍运行在失联的节点上。在 1.5 之后，节点控制器不会强制删除pod信息，直至在集群中确认没有这些 pod。这个情况中， pod 会以 Termiating Unknown状态存在，这些 pod 则需要手动删除。

node 管理

node 在集群中仍是“资源”的形式存在，K8s 会根据资源的描述（如下）检测节点是否处于可用状态

K8s 如果拥有一个 node 的资源，则会不断测试该节点是否可用（启用线程），因此移除节点时，最好删除掉在 K8s 中的 node 资源。

node controller

主要负责 node 的管理：

  • 为节点分配 CIDR 块
  • 维护节点状态，并与云提供商底层状态同步
  • 节点健康检查（40s 超时上报心跳， 5min 超时并开始驱散 pod, 设置检查节点健康间隔时间--node-monitor-period）
